How It Works

The building blocks — the behavior record, its lifecycle, the adapter model, the behavior graph, coverage fingerprints, and the two blast-radius directions. Examples use the dogfooding testbed's passkey-auth spec, SPEC-001, throughout: BEH-001 (login — still proposed), BEH-002 (unit), and BEH-003 (integration).

What changed in the toolkit

If you know the original main, your first question is "what's new?" Two skills are brand-new, three you know grew, and one deliberately learned nothing at all.

Skillvs mainRole in the layer
behavior-runnernewruns accepted behaviors via their adapter and captures TEST → CODE coverage; prints fingerprints as JSON — it never writes the graph. The heavy, tooling-coupled part.
behavior-graphnewowns behavior.json: projects specs, orchestrates the runner, merges fingerprints by trust, and serves Direction A/B.
statusnewthe read-only counterpart of wrap-up: the census + the git-tracked BACKLOG.md.
spec-managerextendedspecs gain behaviors: records; it owns principles.md, decisions/ (ADRs), intents/, and every deterministic gate (verify_links, verify_intent, adr, plus the governance helpers).
wrap-upextendedorchestrates the graded Phase 3.5 pipeline and the two-commit, state-aware staging.
code-graphfixed, not taughtthe substrate. It got the path-alias / cwd / unresolved-signal fix — but stays completely unaware of behaviors.
Why two new skills, not one

They change for different reasons. behavior-runner is the messy part — it boots runners and captures runtime coverage. behavior-graph is pure and unit-testable without ever running a test. And code-graph stays behavior-blind on purpose, so the code substrate remains swappable without touching the behavior layer.

The behavior record

A behavior lives inside a spec's frontmatter, under a behaviors: list. It's a stable, id'd record — not a test file, but a pointer to one. This is exactly what frontmatter.py validates.

knowledge-base/specs/auth/SPEC-001-passkey-login.md
behaviors:
  - behavior_id: BEH-003          # stable across renames
    title: Unknown email does not reveal whether a user exists
    state: accepted          # proposed | confirmed | accepted | quarantined | deprecated
    adapter: cucumber        # required only when state == accepted
    level: integration
    locator: features/auth/passkey-login.feature#unknown-email-does-not-reveal-whether-a-user-exists
FieldRequired?Rule
behavior_idalwaysmatches BEH-NNN; unique within its spec here (cross-spec reuse is caught by verify_links); stable across renames
titlealwaysnon-empty string
statealwaysone of the five lifecycle states
adapteronly when acceptedone of the known adapters — but any typo fails loud in any state
locatoronly when accepted & adapter ≠ manualpoints at the verifying test
leveloptionalunit | component | integration | e2e — the runner's dispatch key
entryoptionalthe route/handler an integration behavior's test drives
Why adapter & locator are conditional

Only accepted asserts a real, linked, passing test — so it's the only state that requires the link. Pre-test states (proposed, confirmed) may omit them — the test is still owed. But if either field is present in any state, it's still validated, so a typo can't silently route a behavior to the wrong runner.

The lifecycle — click a state

A behavior moves proposed → confirmed → accepted, with quarantined/deprecated off to the side. The single most important distinction: only accepted gates wrap-up and gets executed. confirmed means a human agreed the intent is real but the test isn't written yet — advisory, never run.

Inference can only mint proposed

scan/bootstrap produce a review queue of proposed records inside spec frontmatter — never accepted, and never files written into the code tree. A human accepting a candidate is the only thing that promotes it and lets a scaffold or link land as code. Why so strict: intent cannot be reliably inferred from code — auto-generating authoritative-looking tests from the implementation would reintroduce the exact "tests mirror code" problem the layer exists to fix.

Adapters — link, don't re-author

An adapter links a behavior to whatever verifies it. The layer detects a project's tooling instead of hardcoding a stack, and it never forces you to re-author tests you already have.

Author a tagged scaffold (the default for new, user-visible behavior)

spec-manager writes a skeleton .feature — the required tags plus a TODO(scaffold) marker, but no real steps (authoring those is human forward-design work). The @SPEC/@BEH tags are the reverse link.

features/auth/passkey-login.feature
@SPEC-001
Feature: Passkey Login
  # Intent and rationale live in knowledge-base/specs/auth/SPEC-001-passkey-login.md

  @BEH-001
  Scenario: Successful passkey login
    # TODO(scaffold): replace with real steps. Step definitions are not generated.
    Given <initial state>
    When <action>
    Then <expected outcome>

On the testbed this is exactly BEH-001 — a proposed scaffold still awaiting real steps. The TODO(scaffold) marker is load-bearing: an accepted behavior that still carries it is a deterministic error (a lie the gate catches).

Point a locator at a test that already exists

No file is written, nothing is rewritten. You set adapter to the runner and locator to path/to/test#case (or path::node for pytest). The locator is the link — no reverse tag required. This keeps adoption cheap for projects that already have tests.

BEH-002 — linked to an existing vitest test
adapter: vitest
locator: lib/webauthn.test.ts::rejects an expired challenge

Human-verified, no runner

The manual adapter is for behavior that genuinely can't be automated. It carries intent and is skipped by locator resolution — the honest ceiling of what tests can reach, recorded rather than pretended away.

The behavior graph — a sibling, not a schema bump

The graph is behavior.json, generated next to code-graph's graph.json in the git-ignored knowledge-base/.graph/. It's kept separate on purpose: code-graph stays completely unaware of behaviors, so the code substrate can be swapped later without touching the behavior schema.

The one genuinely new edge is TEST → CODE — and it's never asserted as certain. Every edge carries a provenance, in a strict trust order:

sourcemeansconfidencecaptured by
explicithand-declared anchor reserved—not implemented — a future lever
observedthe test actually executed this file0.8vitest V8 coverage (unit)
staticthe file is in the entry's import closure0.5code-graph transitive deps (integration)

Trust order: explicit > observed > static. Static is lower trust because it over-approximates — it lists everything the entry could reach, not what a test did reach. Each edge also carries a freshness commit, so a fingerprint captured before a change to its files can be flagged stale.

Measured on the testbed

Editing lib/webauthn.ts flags exactly the two behaviors that depend on it; editing an unrelated lib flags none; editing the route flags only BEH-003. False-positive rate: 0 on the representative set — the gate is selective, not flag-everything. (Illustrative on a small set — 2 behaviors, 3 changes — not a benchmark.)

Coverage fingerprints — observed vs static

A fingerprint is the set of files a behavior's test exercises. There are two producers, and the difference matters.

Over-approximation is the safe direction

When the layer can't observe exactly which code a test runs (integration on a bundled app), it falls back to the whole transitive import closure of the entry file. Flagging too much is fine — the failure mode you must never have is silently missing touched code. A false "might be affected" just runs an extra test; a false "not affected" misses a regression.

Merge by trust

When the graph is rebuilt, a new run never downgrades good data:

incoming runresult
observedtake it (highest trust)
statictake it — unless the prior edge was observed (don't downgrade)
unknown + test-failedinvalidate — the test is red
unknown + any other reasonpreserve the prior fingerprint

The key insight: a transient "couldn't capture" must not erase a good prior edge. Only a genuine red test invalidates.

Coverage-unknown, never silent

Governance leans on this graph, so it must report when it can't resolve an edge rather than return a small blast radius that looks complete. If a behavior can't be fingerprinted, it is coverage: "unknown" with an empty exercises list — never faked — and it carries a reason so a consumer knows why:

reasonmeaning
level-deferredan adapter/level not implemented yet (e.g. e2e)
test-failedthe test process exited non-zero (red)
no-coveragethe test passed but produced no coverage file
no-entryan integration behavior declares no entry
entry-missingthe declared entry file doesn't exist
no-graphno built code-graph cache
not-runprojected, but the runner emitted nothing for it

The substrate underneath: code-graph

Every blast-radius answer is a projection over a code dependency graph. If that graph is wrong, every governance decision built on it is wrong in the most dangerous way — a small, confident, empty answer that looks complete. So the vision demands a capability contract the substrate must meet before governance can trust it.

The bug dogfooding caught — that a green suite never would

On the first real path-alias project, the graph built cleanly: 229 files, 1052 import edges — and 0 internal edges. Every internal import used the @/ alias, and the resolver dropped non-relative imports as "external." Asking for a route's dependencies returned [] despite three real imports. Worse than incomplete: it returned an empty blast radius as if complete. A git check proved the bug was original since v0.1.0 — dogfooding just exposed it.

The fix taught the resolver to read tsconfig path aliases, anchored resolution to the project (not the process cwd), and — the heart of the contract — tagged every edge internal, external:<pkg>, or unresolved:<imp> so "no dependencies" is distinguishable from "couldn't resolve." The rebuild went from 0 internal edges to 607.

before: 0 internal / 1052 external → after: 607 internal / 488 external / 0 unresolved
