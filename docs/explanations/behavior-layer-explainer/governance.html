Governance

Capturing intent is half the story. Governance is the other half: how captured intent is actually enforced — through six checks, an authority ladder, and one careful rule about when a machine is allowed to block you.

The keystone: block on facts, only acknowledge guesses

Failures are gated by what kind of check produced them — not by a model's self-reported confidence.

deterministic hard-block

A Tier-1 link break, an accepted behavior's test failing, or an accepted-test edit with no intent record. These are facts — wrap-up refuses to complete until they're resolved (fix, declare intent, or quarantine).

model resolve-to-proceed

A model saying "this looks like a contradiction" is a fallible guess. It must be seen and resolved — but never hard-blocks on confidence.

Why not block on the model too?

"A model's 'high certainty' is not a calibrated probability, and blocking on it would train people to rubber-stamp 'declare intent' to escape noise." Once bypass is reflexive, real violations get waved through too. So a model finding surfaces and must be resolved — but confidence becomes a hard gate only after its false-positive rate is measured on a real project and shown acceptable.

G1 — declared-intent records

An accepted behavior's test is its machine-checkable guarantee. The system keeps code, the test, and the description in agreement. Code-vs-test is a fact: you run the test. But that fact-layer has one blind spot — editing the test itself.

The edit-both-and-stay-green hole

If you change the code and its test together, the test still passes — green — so the strongest check in the system sees nothing, even though the guarantee was silently redefined. That is the one way to slip a changed guarantee past every fact-check.

G1 plugs exactly that hole. It gives an accepted test one sanctioned way to change: a durable, machine-checkable INTENT-NNN record. The gate's own question is itself a fact — "was an accepted test edited, and is there a record naming it?" — so G1 stays 100% deterministic and hard-blocks.

Watch the gate fire

The canonical end-to-end case from the G1 design: the anti-enumeration response of BEH-003 is changed.

Control case that proves the gate is narrow: edit a proposed behavior's test → never blocked. Only accepted guarantees are governed. And "chat history saying I meant to" does not count — the record is the source of truth.

G2 — principle enforcement

principles.md is the project's constitution — a short, sharp list of rules above every spec and decision. G2 makes it actually enforced instead of a passive document. "A passive file is not enforcement; these two mechanisms are":

Soft — context injection

The constitution is auto-loaded into the working context of design-time flows and wrap-up, so you build with the rules in view.

Checkpoint — resolve-to-proceed

At wrap-up, the change diff is judged against each principle. A finding must be resolved; wrap-up won't complete while one is open.

The human is the calibration

A "violation" hides two questions: is it real? (the model's fallible judgment) and what should happen? (must be adhered to). Not-hard-blocking answers only the first — it is not a license to let violations slide. "Ignore and push" is not a resolution. Refuting a false positive is a first-class, legitimate resolution — which is exactly what keeps there being no reflexive-bypass pressure.

Three resolutions, one append-only log

ResolutionWhenRecorded as
Fixthe principle really was violatedthe code diff + commit — git is the record, no log entry
Refutethe model flagged a false positivea refuted record in the resolution log
Amendthe principle itself should changeedit principles.md + an amended record

The log — knowledge-base/principle-resolutions.jsonl — is append-only. Its load-bearing job: stop the check from re-nagging about the same refuted false positive on every later change. Retirement is a later superseded record (latest-wins), never a mutated field — so the audit trail is never erased.

LLM-first recurrence triage — fenced

When a finding recurs, the model re-validates the prior against the current code: still clearly the same intentional thing → auto-clear (logged); the code moved so the prior no longer maps → retire & re-judge; the reason no longer excuses what's there → escalate. The guardrails: re-judge against the specific prior reason (not the whole file), bias to escalate on any ambiguity, always log auto-clears, and a finding with no prior always reaches the human — the safety floor.

G3 — contradiction checks

G3 is uniquely intent × intent: when a spec's or ADR's intent is created or changed, does it contradict a higher-authority intent? (G1 is test-vs-record; G2 is code-vs-principle; G3 is intent-vs-intent.) It's a Tier-2 model check — resolve-to-proceed.

Why it's scoped (and stays trusted)

G3 never re-derives the whole repo. A changed spec is compared against principles.md plus the intentional decisions of other specs in the same category — the blast radius of the changed intent. A whole-repo re-derivation on every spec edit would be slow and noisy, and noise erodes trust in the gate.

The authority ladder

Authority order: principle > ADR > spec (with reference/ below all). When two intents collide, the higher-authority one wins by default, so you fix the lower one. Two same-tier items have no automatic winner — that's a consistency conflict to reconcile. Pick a collision:

P4a — ADRs, and why they're always-global

An ADR (ADR-NNN) is a cross-cutting Architecture Decision Record — Decision · Rationale · Rejected Alternatives · Revisit Conditions. Before P4a, knowledge-base/decisions/ was an empty scaffold nothing read. P4a shipped the format and made G3 compare against it. Only accepted ADRs are authoritative.

The big decision: show the model ALL active ADRs, no scoping

Scoping only decides what reaches the model — the model still makes the judgment. Over-scoping (a filter that hides a relevant ADR) is a silent miss, the exact failure this layer exists to prevent, and it's unrecoverable. Under-scoping (an irrelevant ADR reaches the model) is just noise it dismisses in one line. A false negative here is far worse than a false positive — and ADRs are cross-cutting by definition, with a tiny corpus. So the only filter is lifecycle status.

Deterministic adr verify (duplicate ids, dangling supersede links, bad status) does hard-block — it's a Tier-1 integrity fact, like link integrity. The contradiction judgment stays advisory.

P4b — declarative-drift, and why it's the opposite

Declarative intent — a spec's intentional_decisions, a purely-declarative spec's prose, or an accepted ADR's decision — has no test to fail. P4b guards it against the code drifting away, at wrap-up: does the changed code contradict a declared intent that governs it?

Here's the deliberate asymmetry with P4a — and it's correct both ways:

P4a · spec ↔ ADR (G3)P4b · code ↔ declared intent

Honest about its blind spot

Because P4b follows related_code, a declared decision with no related_code is invisible to it. Rather than hide that, drift gaps makes the un-scopable set visible on demand — the recall gap is surfaced, never silent.

Resolve-to-proceed & the append-only logs

G2, G3, and P4b each keep an append-only resolution log (principle-, contradiction-, and drift-resolutions.jsonl). They share four verdicts — refuted, amended, auto-cleared, superseded — and the same discipline: a fix leaves no entry (git records it); a refute/amend/auto-clear is logged; retirement is a later superseded record, latest-wins.

Because all three logs did the same latest-wins / drop-superseded / de-dupe dance, the third copy tipped the balance and the logic was extracted into a shared resolution_log.py — a behavior-preserving refactor proven by the three existing test suites passing with their files byte-unchanged.
