The Journey & The Proof

One guiding rule shaped every sequencing choice: mechanism-first, dogfood-as-you-go. The central behavior loop was validated by running the branch code against a real project as it was built — rather than building horizontal infrastructure or governance ahead of evidence.

Why the order looks the way it does

Phase 0 was deliberately dropped

A separate, staged "vertical proof" in isolation was considered and dropped. Instead, that validation was folded into Phase 1's first real use on the testbed — a conscious risk trade: some early schema, lifecycle, and adapter choices were made provisionally and corrected in contact with reality. (The "F"-numbered friction findings are that correction record.)

Adoption preempted governance

The vision earmarked Phase 3 for governance — but "you cannot govern intent you have not captured." So an adoption track came first, built on one idea: decouple inference from validation. Inference is cheap and done up front (a bootstrap infers hundreds of untrusted candidates); validation is the scarce resource, spent lazily — you only confirm the two or three a change actually touches, in context, at wrap-up.

The phase timeline

Phase 1

Traceability MVP

Formalize spec↔behavior identity and the lifecycle; support Gherkin plus one native adapter; deterministic integrity checks; run accepted behaviors at wrap-up and block only deterministic failures. This is where the Behavior entity, the two adapters, and verify_links are born.

Phase 2

Impact indexing

The two new skills (behavior-runner, behavior-graph) and behavior.json appear. Observed (vitest) + static (code-graph closure) fingerprints, provenance + freshness, and both blast-radius directions. Then the evidence gate: measure false-positives and runtime before governance may trust the graph.

Adoption · SP1–SP5

Getting onto a real codebase

The confirmed state (intent agreed, test owed); greenfield/brownfield bootstrap; validate-on-hit (surface the 2–3 behaviors a change touched); status & a git-tracked BACKLOG.md; and a security↔behavior cross-reference (an accepted behavior is verified evidence that can downgrade a finding).

Phase 3 · G1–G3

Governance

Declared-intent records (G1), principle enforcement (G2), and contradiction checks (G3) — the deterministic-facts-block, model-findings-resolve-to-proceed core.

Phase 4 · P4a/P4b

Expansion: ADRs & drift

ADR support with an ADR-aware, always-global G3 (a research-backed reversal of an earlier category-scoping sketch, to avoid the silent-miss failure mode), and declarative-drift — a blast-radius-scoped, code-anchored guard for untestable intent.

Refactor

The shared resolution-log

Once G2, G3, and P4b each carried a near-verbatim copy of the append-only log logic, the third copy tipped the balance. Two copies had been a deliberate, spec-approved choice — not churning shipped code was worth more than DRY. At three, the logic was extracted into resolution_log.py and the three modules rewritten to delegate. Behavior preservation wasn't argued — it was mechanically guaranteed: the three existing test suites had to pass with their files byte-unchanged (11 + 16 + 19), and they did.

Where it all runs: the wrap-up pipeline

The two-commit pattern

wrap-up splits every run into commit 1 (code) and commit 2 (artifacts) — so the security scan has a stable commit to reference and git history stays clean. The subtle rule: a .feature scaffold is staged by its lifecycle state, not its location — a proposed/TODO scaffold rides the artifacts commit even though it sits in the code tree, and joins the code commit only once it's accepted and its marker is gone.

What dogfooding proved

The testbed was a real, production-shaped Next.js 16 / React 19 / TypeScript app with 307 files and zero tests — deliberately exercising the "no runner" and greenfield-adoption paths. The first natural spec was passkey auth (SPEC-001).

The lie a green suite can't catch

The load-bearing moment

Flipping BEH-001 to accepted while its "test" was still a TODO scaffold produced exactly one error, exit 1, scoped to BEH-001 — its sibling proposed scaffolds in the same file were correctly untouched. The suite is green (nothing runs); the layer says exit 1. That's a lie about a guarantee that a green test suite is structurally incapable of catching.

The latent bug it exposed

The testbed used the @/ path alias pervasively — a direct stress test of the code substrate. It exposed that impact analysis had been returning an empty-but-complete-looking blast radius for every path-alias project (a git diff showed the resolver had been unchanged since v0.1.0 — only a 2-line path rename — so the bug was original, not introduced). Fixing it — alias resolution, cwd-independence, and an honest unresolved: signal — took the graph from 0 to 607 internal edges. This is precisely why "coverage-unknown, never silent" is a non-negotiable.

The false green that reshaped the design

The original plan for integration behaviors was to import the framework route handler in-process and call it directly. On an ordinary CommonJS-default project, Node's require(esm) rules forbid it — and flipping the app to "type": "module" made the test pass, which is exactly the trap: a false green, bought by demanding every adopter restructure their app. The resolution became a principle: a behavior test drives the app over its real interface (HTTP) against a running instance — never by importing internals.

Observed coverage of that running app then captured only framework internals — zero app code (bundling plus a separate render worker). Rather than ship a framework-specific coverage dance, observed integration coverage was deferred (the parked per-framework V8+CDP adapter) and the static import-closure became the honest baseline — over-approximating, in the safe direction.

Selective and fast

Editing lib/webauthn.ts flags exactly the two behaviors that depend on it; an unrelated lib flags none; the route flags only BEH-003. Incremental --check on a change touching no exercised code is 0.07 s (zero re-runs) versus a full build at ~1.4–2.4 s.

Read the numbers honestly

The FP=0 / 0.07 s figures come from a small set — 2 behaviors, 3 changes. They validate the mechanism, not trustworthiness-at-scale. That's exactly why fingerprint-driven governance stays advisory; only a real test-failed of an accepted behavior blocks.

The "flood" question, answered

Does inferring behaviors for a whole codebase drown the engineer? A full brownfield scan (a coordinator + 7 parallel agents over ~224 files) produced ~383 candidate behaviors — a flood if reviewed eagerly, fine because nothing is. Each feature area was individually manageable (35–63), the executable-vs-declarative split (~88/12) was sensible, and the one-time cost was ~260k tokens / ~65 s.

Why this works

The scan is a manageable backlog, not a flood, because review is lazy: validate-on-hit surfaces only the handful a change touches, and worklists grind the cold tail one at a time. That makes the adoption track load-bearing, not optional — it's the only thing that makes a ~380-candidate corpus tractable.

How it gets onto a real codebase

The adoption track (SP1–SP5) is a four-step arc, all built on the same idea: inference is cheap, validation is the scarce resource — so spend it lazily, in the flow of work.

  1. Day one — bootstrap. A deterministic detector classifies the project by internal import edges — real feature wiring, not file count (a boilerplate-heavy scaffold looks greenfield despite its many files). The evidence is always shown and the recommendation is overridable. Brownfield → a scan mints proposed candidates as records only; greenfield → skip inference and author behaviors forward as you build.
  2. In the flow — validate-on-hit. Each wrap-up surfaces just the proposed/confirmed behaviors this change touched, re-inferred against the current code — confirm, edit-then-confirm, or skip. Touched code with no covering behavior gets a "capture one?" prompt. Advisory, bounded, skippable.
  3. The cold tail — status & the backlog. A generated, git-tracked BACKLOG.md ("do not edit — run status to refresh") shows the census: behaviors to confirm, tests owed, coverage gaps, open findings — it diffs in PRs, so the team sees outstanding work without running anything. Two worklists (review intent, review tests) grind the never-touched tail one item at a time.
  4. The payoff — security cross-reference. Once a behavior is accepted, its passing test is the strongest possible "intentional" evidence a security scan can cite — a verified guarantee, not a prose claim.
The payoff, concretely

A scan flags "endpoint doesn't verify the user exists" (SEC-001). BEH-003's passing test proves the uniform response is the intended anti-enumeration guarantee, so the finding is downgraded to intentional with behavior_ref: BEH-003 — annotated, never deleted. SEC-002, which no accepted behavior covers, stays open. Only accepted qualifies; a confirmed behavior at most adds an advisory note — the finding stays open until a test verifies it.

What's deliberately parked

Deferral here is a quality decision, not neglect — the recurring note is "do it properly, as its own effort." Each parked item records what, why deferred, and how to pick it up.

Remaining Phase-4 leaves

P4c — more adapters deprioritized

Only the vitest unit runner is actually implemented; jest/pytest/playwright/… are allow-listed but stubbed. Deprioritized because it doesn't serve the current stack. If wanted: jest is near-free (same coverage format); pytest adds Python. Observed e2e is not P4c — it needs the deferred V8+CDP adapter.

P4d — calibrated enforcement evidence-gated

Promoting a model finding to a hard gate — but only after its false-positive rate is measured on a real project and shown acceptable. That evidence doesn't exist yet, so it waits by design.

Deferred capabilities (each its own effort)

  • Per-framework observed-coverage adapter (V8 + CDP) — "the big one." Real observed integration/e2e coverage by launching the app under --inspect and collecting V8 coverage over the Chrome DevTools Protocol, then remapping bundled output to source. Framework-specific; static-via-code-graph is the shipped baseline meanwhile.
  • Graphify substrate fallback — the homegrown resolver was fixed, so "don't adopt a dependency we don't need yet." Revisit if it keeps accruing edge cases (known gaps: extends chains, per-edge confidence, monorepos).
  • Full E2E for the happy-path login (BEH-001) — needs a signed WebAuthn assertion (Playwright + virtual authenticator). Stays proposed until then.
  • Vendor-substrate ownership boundary — "the strongest unaddressed adoption case": you own a thin layer over a large vendor codebase you must not touch. Needs a first-class ownership boundary so a naive scan of a vendor checkout doesn't turn ~383 candidates into tens of thousands.

Phase 5 — the new frontier (work-laptop enablement)

The layer is functionally complete for the greenfield/TypeScript case. The next frontier is using it on a polyglot enterprise stack from VS Code + GitHub Copilot — two independent tracks:

Track A — multi-agent portability

The deterministic scripts are already portable stdlib-Python CLIs; only the orchestration layer is Claude-specific. Make scripts self-locating, add per-agent instruction files + an installer, and consolidate on AGENTS.md and/or MCP. Self-contained, medium effort.

Track B — polyglot code-graph

code-graph is a homegrown TS/JS import resolver — blind to Java, Docker, Helm, and config-as-code. Real languages need a real parser, and config-as-code needs a second "resource graph" of reference/deploy edges. The true blocker, and the largest effort.

The pivotal fork — decide first

The plugin's north star is "stdlib-only Python, zero-install." Java / tree-sitter / graphify break that. The fork: keep zero-install (homegrown per-language resolvers — limited, brittle) vs adopt a dependency (real multi-language, gives up zero-install). It's the one open decision that gates the largest remaining effort.

The honest status

  1. The original vision is essentially delivered. Phases 1–4 are shipped, the ADR machinery is in, and the substrate capability contract is largely satisfied (graphify still in reserve). What remains are deliberately parked leaves and the new Phase-5 frontier.
  2. The branch is unmerged by a standing decision. All the work is local on feat/behavior-layer; the operational parking items (revert the dev symlink, publish the real release) are explicitly "do before any release." The layer is functionally complete for the greenfield/TypeScript case, but hasn't been cut into a release.
