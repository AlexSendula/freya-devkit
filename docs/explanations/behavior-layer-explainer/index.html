freya-devkit · the behavior layer

Your tests prove what your code does. Nothing proves what it's supposed to do.

freya-devkit was a strong reverse-sync engine: after you change code, it re-syncs a dependency graph, docs, specs, and security findings. What it couldn't do was capture what the system is supposed to do — in a form that is executable, intent-driven, and visible at design time. The Behavior Layer adds exactly that.

The problem: four ways behavior goes unguarded

Every failure below is a "before" — the reason the whole initiative exists. All four come straight from the vision document.

🔁

Tests mirror code, not intent

Tests are usually written from the code, so they verify "what the code does," not "what it should do." Coverage looks fine while the actual intended behavior goes unguarded.

👻

Behavior drifts silently

A change can alter behavior unintentionally and still pass a green suite, because nothing pins the intended behavior. The suite never turns red.

📄

Specs are inert

spec-manager captured why well, but the "what" was an acceptance-criteria checklist that nothing executes — verify could only eyeball it.

🎯

No behavioral impact analysis

code-graph answers "what code does this change touch?" Nothing answered "what intended behavior does this change touch?"

The through-line

A green, high-coverage suite can be perfectly consistent with the system doing the wrong thing — because the tests were derived from the code rather than from intent. Fixing that is the entire point.

The one idea

Make intended behavior a first-class, executable, blast-radius-aware artifact.

Concretely: a behavior becomes a stable, id'd record — BEH-003 — with a lifecycle, linked through a runner-agnostic adapter to a real, executable test. The test is authored as Gherkin where Given/When/Then adds intent-clarity, or linked to an existing native test (jest, pytest, playwright…) where one already covers the behavior. Then every behavior is wired into a behavior graph so any change can be traced to the behaviors it touches.

Not all "intent" is the same thing

The layer sorts intent into four kinds — each with a home and a way it's verified. This taxonomy is what keeps an executable guarantee from being confused with a prose decision.

Kind of intentExampleHomeVerified by
Executable behavior"a new user can register a passkey and is logged in"a Behavior linked to a test, in the code treethe runner, via the adapter — automated regression
Declarative decision (feature-local)"this feature offers no password fallback"a spec's Intentional Design Decisionshuman review + certainty
Cross-cutting decision (ADR)"we use Postgres, not Mongo"knowledge-base/decisions/human review + certainty
Principle (project-wide)"every endpoint authenticated by default"knowledge-base/principles.mdapplies to all of the above
The promotion rule

Prefer turning a declarative decision into a guard scenario (often a negative test) when it's cheap — "no password fallback" becomes When credentials are POSTed to /login, Then it is rejected. When a decision genuinely can't be tested, it stays declarative. That's the honest ceiling of what tests can do — not a failure.

The five non-negotiables

These are the load-bearing constraints. Get them right and the rest of the design follows.

  1. Accepted behavior is authoritative — but classify, don't just fail. An accepted behavior's failure blocks completion until it is classified — a regression (fix the code), an intended change (record it), or a test-infra failure (quarantine). It deliberately does not claim every red test means broken code — only that an accepted behavior may not fail silently or unclassified.
  2. Intent has two kinds. Executable behavior → an executable test. Declarative decisions → prose, governed by review. Only the first becomes a test.
  3. Truth flows in two directions. Specs + behavior are forward / authoritative (code conforms to them). Reference docs are reverse / descriptive (they mirror the code). Never conflate the two.
  4. Single source of truth, never a duplicate. A generated projection (one source, a derived view) is allowed; a hand-maintained duplicate (two editable copies that drift) is forbidden.
  5. Adapter-based, runner-agnostic. A behavior is modelled independently of any runner. No hardcoded stack — and no forced re-authoring of tests a project already has.

What it looks like end to end

A behavior moves through a lifecycle, and only one state is authoritative:

proposed → confirmed → accepted · quarantined deprecated

proposed = a candidate (often inferred), intent not yet human-confirmed. confirmed = a human agreed the intent is real, but the test is still owed. accepted = the test exists and passes — only accepted behaviors gate and get run. quarantined/deprecated sit outside the authoritative set.

That executable link is also what fixes problem #3: verify is upgraded from eyeballing prose to actually running the linked tests.

With behaviors linked to code, blast radius flows two ways:

Direction A — code → behavior

Regression early-warning: git diff → code-graph blast radius → intersect with test fingerprints → "these behaviors exercise code you're touching." At wrap-up it re-runs only the affected accepted behaviors.

Direction B — intent → behavior

The planning question: a spec/behavior edit → its affected behaviors → the implementing code. Available at design time: "this change touches behaviors X, Y — change or preserve?"

How It Works → unpacks the graph, the coverage fingerprints, and both directions interactively.

Turning captured intent into a gate

Capturing intent is half the story; the other half is enforcement. The keystone is a single, careful distinction:

Deterministic facts hard-block

A broken link, an accepted test failing, an accepted-test edit with no intent record — these are facts. Wrap-up refuses to finish until they're resolved.

Model findings resolve-to-proceed

A model saying "this looks like a contradiction" is a fallible guess. It must be seen and resolved — but never hard-blocks on confidence, because blocking on uncalibrated "high confidence" just trains people to rubber-stamp past it.

The gates themselves: Tier-1 link + ADR integrity and the declared-intent gate block on facts — as does the gated run of affected accepted behaviors — while principle, contradiction (spec/ADR), and declarative-drift checks surface model findings you resolve-to-proceed ("ignore and push" is not a resolution). Governance → walks each one, interactively.

Why not just adopt spec-kit?

Two ideas were borrowed from github/spec-kit; the engine was not.

  • Borrowed — a constitution. spec-kit's constitution.md becomes knowledge-base/principles.md: project-wide rules above all specs and decisions.
  • Borrowed — executable acceptance criteria. spec-kit writes Given/When/Then but leaves it inert prose. The Behavior Layer makes the same grammar executable — via a Gherkin .feature or a linked native test.
  • Not adopted — spec-kit wholesale. spec-kit is a forward build engine (specify → plan → tasks → implement). freya-devkit already has a forward flow, and its differentiator is the reverse-sync + intentional-decisions/security tie-in. Adopting spec-kit wholesale would discard that.

The honest status

Delivered & dogfooded

The layer shipped across a 135-commit branch and was dogfooded on a real, production-shaped Next.js / TypeScript app with zero pre-existing tests. Phases 1–4 of the original vision are all in: traceability, impact indexing, the adoption track, governance (G1–G3), ADR support (P4a), and declarative-drift (P4b).

Scope & honesty

It is functionally complete for the greenfield / TypeScript case. Remaining Phase-4 leaves are deliberate deferrals (more adapters deprioritized; calibrated enforcement evidence-gated). The branch is unmerged by a standing decision, and the new frontier — multi-agent portability + a polyglot code-graph — is a fresh "Phase 5."
