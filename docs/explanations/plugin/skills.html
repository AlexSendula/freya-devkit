The Skills

All ten, in depth

Each skill does one thing and composes through the shared knowledge-base/. Expand any card for its purpose, key commands, what it reads and writes, and its honest limits. Full CLI flags live on the Reference page.

Script-backed vs prose-driven

Some skills ship a real, stdlib-only Python engine that does deterministic work (code-graph, spec-manager, behavior-graph, behavior-runner, status; docs-manager has one detection helper). Others are prose-driven — the skill is instructions the agent follows, with no binary (codebase-security-scan aside from its audit workflow, codebase-security-resolver, dependency-vulnerability-check, wrap-up). The badge on each card tells you which.

1 · Foundation

code-graph Foundation Python engine The dependency graph + blast radius every other skill queries.

Builds a reverse-dependency graph by parsing import/export relationships (regex, not a full parser) for TypeScript/JS, Python, and Go. It answers "what depends on this?" so consumers can process a change's whole blast radius, not just the file that changed.

Key commands

/freya-devkit:code-graph build          # full scan → graph.json
/freya-devkit:code-graph update         # incremental via git diff
/freya-devkit:code-graph impact <file>   # blast radius (accepts many files)
/freya-devkit:code-graph query <file>    # dependencies + usages
/freya-devkit:code-graph dependents <file>
/freya-devkit:code-graph dependencies <file>

Reads / writes

Reads your source + git. Writes knowledge-base/.graph/graph.json and classifications.json — a regenerable, git-ignored cache. On build it classifies directories (rules → AI → user); non-interactively, uncertain directories default to source so real code is never silently dropped.

Composes with

Everything. docs-manager, spec-manager, the security scan, and behavior-graph query it for impact; each degrades to a plain git diff when it's absent.

Good to know

  • Regex, not AST — dynamic import(variable) may be missed; comments/strings can mismatch.
  • External packages are recorded as external: leaves, not traversed.
  • Go / absolute-Python intra-project imports often fall through to external:; internal-edge resolution is strongest for TS/JS relative + aliased imports.
  • Monorepos want one graph per subproject; tsconfig extends is not followed.

2 · Knowledge

docs-manager Knowledge Prose + 1 helper Standardized project docs, impact-aware, in knowledge-base/reference/.

Creates and maintains a standard documentation set with a coordinator + parallel workers architecture: one coordinator detects the stack and gathers business context, then one worker per doc type (ARCHITECTURE, API, DATABASE, DEPLOYMENT, …) runs in parallel. Undetectable details become [TODO:] placeholders resolved later in batched questions.

Key commands

/freya-devkit:docs-manager init            # create the doc set
/freya-devkit:docs-manager update          # impact-aware refresh
/freya-devkit:docs-manager update api      # one doc
/freya-devkit:docs-manager review | sync | resolve | upgrade-diagrams

Reads / writes

Reads code + code-graph impact. Owns knowledge-base/README.md (the index) and reference/*.md. Its only real code is detect_project.py (stack + test-runner detection).

Composes with

code-graph (impact-aware update, degrades to git diff); wrap-up runs it as a step. Its reference/ docs give the security scan architectural context.

Good to know

  • Almost entirely prose-driven — doc quality depends on the model following the instructions.
  • Certainty scoring is not a docs-manager feature — that's spec-manager. Its analog is [TODO:] markers and a ✅/⚠️/❌ review.
  • Some templates/evals still reference an older docs/ layout; knowledge-base/ is authoritative.
spec-manager Knowledge Python engine Specs, intentional-design decisions, the behavior lifecycle, and governance.

The intent owner. It captures what a feature does and why it was designed that way, with special emphasis on intentional design decisions that might look like bugs. It owns four things in the knowledge base and hosts the behavior lifecycle plus the deterministic halves of governance.

The false-positive filter

A spec's Intentional Design Decisions section is what a security scan reads to tell a real vulnerability from an on-purpose choice ("No password fallback — a scanner flagging missing password auth should be ignored").

Key commands

/freya-devkit:spec-manager bootstrap       # one-time onboarding (see Get Started)
/freya-devkit:spec-manager create <name>   # author intent forward (certainty 100)
/freya-devkit:spec-manager scan            # infer proposed candidates (brownfield)
/freya-devkit:spec-manager update          # git-aware incremental sync
/freya-devkit:spec-manager verify          # deterministic integrity checks
/freya-devkit:spec-manager intent new <BEH>  ·  adr create|list|verify

Reads / writes

Reads code-graph impact + reference/ docs. Writes specs/ (by category), principles.md (the constitution), decisions/ (ADRs), intents/ (INTENT-NNN), and the three *-resolutions.jsonl governance logs.

Composes with

The security scan reads its intentional decisions; behavior-graph and behavior-runner parse its behavior records; wrap-up runs its governance scripts; status reads its frontmatter for the backlog census.

Good to know

  • scan only ever produces proposed candidates — never accepted, never files in the code tree. Acceptance is a human act.
  • Authority order for contradictions: principle > ADR > spec.
  • certainty (0–100) gates review of inferred specs; executable-behavior trust is the lifecycle state, not the number. See the Behavior Layer.
behavior-graph Knowledge Python engine Owns behavior.json; answers both blast-radius directions.

The graph layer of the behavior layer. It projects your specs' behavior records and merges in coverage from behavior-runner to build behavior.json — the BEHAVIOR → TEST → CODE projection — then answers impact questions both ways.

Key commands

python "$CLAUDE_PLUGIN_ROOT/skills/behavior-graph/scripts/behavior_graph.py" \
  --build --project .                    # refresh behavior.json
  --affected <file...>                    # Direction A: code → behaviors
  --implements <BEH-NNN>                  # Direction B: behavior → code
  --check --base <commit>                 # regression gate (wrap-up)
  --surface --base <commit>  ·  --gaps  ·  --covering <file>

Reads / writes

Reads graph.json + spec frontmatter + runner fingerprints. Writes knowledge-base/.graph/behavior.json (git-ignored). Merges by trust: observed > static; a red test invalidates, any other unknown preserves prior coverage.

Composes with

Queries code-graph (impact) and behavior-runner (coverage) — code-graph never learns about behaviors. Consumed by wrap-up (--check / --surface), status (--gaps), and the security scan (--covering).

Good to know

  • Only accepted + confirmed behaviors are projected; proposed lives only in specs. (See the Behavior Layer for the lifecycle.)
  • Only accepted behaviors can gate a regression check.
  • Queries reflect the last --build snapshot — re-build after spec/code changes. (Its SKILL.md documents 3 modes; the script implements 7.)
behavior-runner Knowledge Python engine Runs accepted behaviors; emits observed coverage fingerprints.

A producer only: it runs accepted, non-quarantined behaviors through their adapter and prints TEST → CODE coverage fingerprints as JSON. It never writes behavior.json — that's behavior-graph's job.

Key commands

python "$CLAUDE_PLUGIN_ROOT/skills/behavior-runner/scripts/run_behaviors.py" \
  --project . --level unit --emit-fingerprints
  --states accepted confirmed            # default: accepted only
  --list                                 # list matching behaviors, don't run

Coverage model

ValueConfidenceHow
observed0.8Runtime V8/istanbul coverage (vitest unit)
static0.5code-graph closure of a declared entry (integration)
unknown—No usable coverage — usually carries a reason, never faked

Good to know

  • Only the vitest unit path is implemented; jest/other adapters return unknown · level-deferred. Integration coverage is static-only; e2e is a later plan.
  • The vitest invocation is hardcoded to pnpm.
  • confirmed behaviors are never executed (test owed) — so they can never be test-failed and never gate. (See the Behavior Layer for the lifecycle.)

3 · Analysis

codebase-security-scan Analysis Prose + audit workflow Six parallel finders + adversarial verification; specs cut false positives.

Application-code security auditing. Six specialized agents scan in parallel (Auth/Authz; Injection; Secrets; API/Network; Config/Deps; File/Resource), findings are validated online and adversarially refuted to kill false positives, and each finding's lifecycle is tracked across scans.

Key commands

/freya-devkit:codebase-security-scan scan       # full
/freya-devkit:codebase-security-scan update     # incremental (used by wrap-up)
/freya-devkit:codebase-security-scan audit      # exhaustive, Workflow-powered, on-demand
/freya-devkit:codebase-security-scan check-specs · impact <file>

Reads / writes

Reads reference/ docs + specs/ + accepted behaviors + code-graph. Writes a dated prose report (security/codebase-security/YYYY-MM-DD.md), a machine-readable findings.json, and .security-last-scan.

Good to know

  • Only accepted behaviors downgrade a finding — a verified guarantee beats a prose spec; proposed/confirmed add only an advisory note.
  • Standard verification runs 2–3 refutation lenses per finding; only a unanimous refute drops one. A split stays needs review.
  • audit is deliberately not wired into wrap-up — it's a background multi-agent Workflow for pre-release.
dependency-vulnerability-check Analysis Prose-driven Supply-chain CVE audit for Node projects.

The supply-chain half of security (its sibling scan covers your code). It detects the package manager from the lock file, runs <pm> audit --json, then enriches and validates each finding with web searches to strip false positives and note applicability.

Key commands

/freya-devkit:dependency-vulnerability-check
# schedule it:
/loop 1w /freya-devkit:dependency-vulnerability-check

Reads / writes

Reads lock files + package.json + WebSearch. Writes dated reports to knowledge-base/security/dependency-vulnerabilities/YYYY-MM-DD.md (preserved per date, not overwritten).

Good to know

  • Node / npm ecosystem only (npm, yarn, pnpm) — no pip, cargo, go modules.
  • Reports, doesn't fix — it suggests npm audit fix and updates.
  • Complementary to the code scan, but not invoked by wrap-up.

4 · Orchestration

wrap-up Orchestration Prose orchestrator Runs the whole team in order under a two-commit pattern.

The post-implementation orchestrator. After you implement a feature, it runs each skill in dependency order and lands a clean two commits: your code, then all generated artifacts.

The pipeline

0.  commit code (if uncommitted)
1.  code-graph update
2.  docs-manager update
3.  spec-manager update
3.5 behavior integrity + governance gates   # see Governance
4.  codebase-security-scan update            # never audit
5.  commit artifacts (+ refresh BACKLOG.md)

Key commands

/freya-devkit:wrap-up
/freya-devkit:wrap-up "feat: add passkey login"
/freya-devkit:wrap-up --no-security --no-docs   # skip steps

Good to know

  • A behavior scaffold's commit class follows its lifecycle state, not its file location — a proposed .feature scaffold (a placeholder test with no real steps yet) rides the artifacts commit; an accepted test rides the code commit.
  • Phase 3.5 is where deterministic checks hard-block and model judgment resolves-to-proceed. "Ignore and push is not a resolution." See Governance.
  • It warns and skips any missing skill rather than failing.
status Orchestration Python engine Read-only backlog: what intent, tests, and findings are outstanding.

The read-only check-counterpart of wrap-up. Where wrap-up does and syncs, status only reports — it aggregates outstanding behavior, coverage, and security work into one summary and, on request, regenerates the git-tracked BACKLOG.md.

Key commands

python "$CLAUDE_PLUGIN_ROOT/skills/status/scripts/collect_status.py" \
  --project . --format text --write-backlog
# agent-driven worklists:
status review intent   # proposed → confirm, one at a time
status review tests    # confirmed → link/write a test → accept
status gaps            # whole-repo uncovered code

Reads / writes

Reads specs/ frontmatter, behavior.json, findings.json, and verify_links. Writes only knowledge-base/BACKLOG.md. Every source degrades to a note independently — the command never blocks or crashes.

Good to know

  • This drains the "cold tail" — behaviors no work ever touches, so wrap-up's validate-on-hit never surfaces them.
  • The intent worklist is sorted lowest-certainty first, so the least-certain proposals surface for confirmation first.
  • It never auto-authors a test — that's the engineer's work.

5 · Resolution

codebase-security-resolver Resolution Prose-driven Interactive fixer; routes every finding to a terminal state.

The actor half of the security loop. It reads an existing report and drives an interactive workflow — list → select → validate → confirm → plan → implement → commit code → hand off to wrap-up. Its signature feature is metadata cleanup: every skipped finding still gets routed to a terminal state, so skipping is never a no-op.

Key commands

/freya-devkit:codebase-security-resolver            # interactive
/freya-devkit:codebase-security-resolver fix SEC-001 SEC-003
/freya-devkit:codebase-security-resolver fix --critical   # --high, --medium are cumulative
/freya-devkit:codebase-security-resolver fix --dry-run

Terminal states for a skipped finding

CategoryBecomes
Already resolvedreport → RESOLVED
Intentional designspec entry added + report → INTENTIONAL
Outdated (removed code)report → OBSOLETE / removed
Misclassified (not security)report → INFO (downgraded)

Good to know

  • Uses code-graph for blast radius; if a finding's file has 0 dependents it recommends deleting the dead code over fixing it.
  • Commits code first (fix(security): …), then hands off to wrap-up for the artifacts commit — the two-commit pattern.
  • Prose-only: the tables and ✅/⚠️ lines are templates the agent reproduces, not program output.
