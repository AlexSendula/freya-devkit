Get Started

Stand it up on your project

One command brings the whole toolkit up. It detects whether your project is brand-new (greenfield) or already substantial (brownfield) and takes the right path — and it always lets you confirm or override.

Install

  in Claude Code
/plugin marketplace add AlexSendula/freya-devkit
/plugin install freya-devkit@freya-devkit

Then every skill is invoked with the plugin namespace, e.g. /freya-devkit:code-graph help. The freya-devkit@freya-devkit string is <plugin>@<marketplace> — the repo hosts its own single-plugin marketplace, so both names match. For local development, point the marketplace at a checkout path instead.

The one entry point: bootstrap

spec-manager bootstrap is the unified "bring the plugin up on this project" flow. It replaces running init / code-graph build / scan by hand, and it's one-time — after it, everything is incremental.

  1. Init structure — creates the knowledge-base/ layout and principles.md (idempotent; never clobbers existing files).
  2. Build the code graph — runs code-graph build; the shape detector needs it.
  3. Detect shape and recommend — shows you the recommendation and its evidence, then asks you to confirm or override.
  4. Branch — greenfield or brownfield (below).
  5. Summary — reports what was created; for brownfield, a count of proposed candidates, with the reminder that nothing needs review now.

The shape detector

The greenfield/brownfield call is objective and transparent: it counts internal import edges in the code graph — real feature wiring — not raw file count. A bare scaffold can have hundreds of boilerplate files and zero real wiring. Try it:

Set edges to 0 → greenfield; any positive value → brownfield. (No graph at all → unknown: run code-graph build first.) The detector never forces a branch; a human confirms on sight — which is the escape hatch for, say, a large Rust codebase that code-graph can't yet wire.

Your path

There's no code to infer intent from, so bootstrap skips inference and you author intent forward as you build. This is the easier, intent-first path.

  1. Bootstrap — /freya-devkit:spec-manager bootstrap detects greenfield, skips scan, and builds an (empty) behavior graph. It prints: "Greenfield project — no inference run. Author behaviors forward as you build with spec-manager create."
  2. Scaffold docs — /freya-devkit:docs-manager init. On a near-empty project many docs are [TODO:] placeholders you fill in as it grows.
  3. Author intent — /freya-devkit:spec-manager create <feature> for each feature as you build it. New specs are certainty 100; add behavior records for observable behavior.
  4. Implement + link a test — write (or link) the test; once it passes, bump the behavior to accepted. Acceptance is a byproduct of implementing, not a separate ceremony.
  5. Wrap up — /freya-devkit:wrap-up. Two commits: code, then artifacts.

There's real code with intent that was never written down. Bootstrap reverse-engineers a review queue of proposed candidates — never authoritative, never in the code tree.

  1. Bootstrap — /freya-devkit:spec-manager bootstrap detects brownfield and runs scan to infer candidate behaviors (warns first — it spawns discovery agents and can take a while), then builds the behavior graph.
  2. Docs from the code — /freya-devkit:docs-manager init (a separate step — bootstrap doesn't run docs-manager). Docs give the security scan and spec inference architectural context.
  3. Security baseline — /freya-devkit:codebase-security-scan scan. The first run has no prior report; it becomes the baseline.
  4. Drain the queue lazily — nothing needs review immediately. Work the tail with /freya-devkit:status review intent (proposed → confirm) and review tests (confirmed → accept), or let wrap-up's validate-on-hit surface behaviors as you touch their code.
  5. Go incremental — /freya-devkit:wrap-up after each feature, /freya-devkit:status to check where you stand.

A fresh brownfield behavior.json is near-empty by design — scan only produces proposed records, and the graph projects only accepted/confirmed. That's correct, not a failure.

After onboarding: incremental forever

Bootstrap is a one-time act. From then on, the loop is small:

wrap-up

After each feature: refresh graph → docs → specs → behavior integrity & governance → security, then two commits. The mutating, syncing path.

status

Any time: a read-only census of outstanding intent, tests owed, coverage gaps, and open findings, refreshing BACKLOG.md. The checking path.

Why bootstrap-first matters

Incremental update assumes a prior sync. On a never-synced project, wrap-up refuses to silently run a full-codebase generation (the never-synced guard). Running bootstrap first — which builds the graph and initializes the structure — is exactly what makes your first wrap-up behave.
