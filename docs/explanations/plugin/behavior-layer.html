The Behavior Layer

Intended behavior, made first-class

The newest capability. Ordinary tests mirror what the code does; behavior can drift while a green, high-coverage suite stays green. The behavior layer fixes that by making intended behavior a first-class, executable, blast-radius-aware, governed artifact.

The one idea

A BEH-NNN record captures an intended behavior, carries a lifecycle, and is linked through a real test down to the code it exercises. Now you can ask "what intended behavior does this change touch?" and "is this behavior still verified?" — questions a plain test suite can't answer.

Four kinds of intent

Not all intent is a testable behavior. The toolkit distinguishes four kinds, in a clear authority order:

KindWhat it isWhere it lives
Executable behaviorAn observable behavior a test can verify.a BEH-NNN record in a spec
Declarative decisionA feature-local choice with no test (e.g. "no password fallback").a spec's Intentional Design Decisions
ADRA cross-cutting architecture decision.decisions/ADR-NNN.md
PrincipleA project-wide rule — the constitution.principles.md

Authority for conflicts runs principle > ADR > spec. How each is enforced is the subject of the Governance page.

The SPEC → BEHAVIOR → TEST → CODE chain

A behavior links four things. That chain is what lets impact flow in both directions.

The lifecycle

Trust in a behavior is not a number — it's the lifecycle state. Click a state to see what it means and whether it gates.

Forward path: proposed → confirmed → accepted. quarantined and deprecated are side states. Only accepted is executed and can block a change.

Adapters: linking a behavior to a test

A behavior becomes verifiable through an adapter. There are two ways, and the toolkit never writes real test steps — that's human forward-design work.

The behavior graph & its two directions

behavior-graph owns behavior.json — a generated projection sibling to graph.json. Because it links behaviors to the code they exercise, it answers blast radius both ways.

Coverage fingerprints

When a behavior is run, the runner records which code the test actually executed — a fingerprint. Coverage is one of three honest values, and "unknown" always carries a reason (it's never faked):

CoverageConfidenceSource
observed0.8Real runtime V8/istanbul coverage — the test ran and we watched (vitest unit).
static0.5code-graph's transitive closure of a declared entry — analysis, no execution (integration).
unknown—No usable coverage — carries a reason, never attributed to code falsely.

Why coverage is "unknown"

reasonMeaning
level-deferredAdapter/level not yet implemented (only vitest unit is).
test-failedThe test process exited non-zero — this one invalidates prior coverage.
no-coveragePassed but produced no coverage file.
no-entryIntegration behavior has no entry declared.
entry-missingThe declared entry file doesn't exist.
no-graphNo code-graph cache — run code-graph build.
not-runThe graph-side default for a projected behavior the runner didn't fingerprint this build.
Merge by trust

Rebuilding never falsely loses coverage. observed > static: a static run won't overwrite a prior observed edge. A test-failed unknown invalidates the fingerprint (the test is red); any other unknown preserves whatever was there before.
