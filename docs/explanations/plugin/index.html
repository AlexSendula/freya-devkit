The whole plugin, explained

A knowledge base that stays in sync with your code.

freya-devkit is a Claude Code plugin of ten composable skills. Each keeps one kind of artifact — your dependency graph, docs, feature specs, intended behaviors, and security posture — coherent with the code as it changes. They cooperate through one shared, version-controlled knowledge-base/ folder, and one command runs the whole team.

The problem it solves

As code changes, the things around the code rot: docs go stale, design intent is forgotten, security findings drift, and tests quietly start mirroring what the code does instead of what it should do. Doing all that upkeep by hand, in the right order, every time — nobody does.

🧱

The monolithic prompt

One giant instruction blob bloats context, resists change, and can't be mixed and matched. freya-devkit replaces it with focused skills that load only when needed and compose on demand.

📉

Artifacts drift from code

Docs, specs, and security reports fall out of date the moment code changes — and no one remembers to update all of them, in order. Each skill re-syncs its own artifact incrementally.

💥

Change-impact blindness

Updating only the files you literally edited misses the blast radius — the dependent files a change actually reaches. A dependency graph turns "changed" into "affected."

🔧

Intentional design gets "fixed"

Scanners and agents flag deliberate decisions as bugs because nothing records that a behavior is on purpose. Specs capture the decision so the scan leaves it alone.

🎯

AI output treated as certain

Without a confidence signal you can't tell which generated content to trust versus review. Inferred specs carry a 0–100 certainty so uncertainty is visible, not hidden.

🫥

Behavior goes unguarded

Tests mirror what code does, not what it should. Behavior can drift while a green, high-coverage suite stays green. The behavior layer makes intended behavior a first-class, executable, gated artifact.

The one idea

Derive one shared source of truth from the code — a dependency graph — and let a team of single-purpose skills each keep one artifact coherent by reasoning about blast radius instead of just changed files, scoring their own confidence, and degrading gracefully when a teammate is missing.

The mental model

Think of the skills as specialized team members working on your codebase: an analyst (code-graph), a writer (docs-manager), an architect (spec-manager), auditors (the security skills), and an integrator (wrap-up). Each has a role, knows its job, and collaborates through the shared knowledge base.

Five core beliefs

  1. Intentional design is real. Code isn't just "correct" or "incorrect" — some things are on purpose. Specs record those decisions so tools stop trying to fix them.
  2. Acknowledge uncertainty. AI-generated content carries a certainty score rather than pretending to be sure. Low confidence asks for review; high confidence is trusted.
  3. Impact awareness. A change isn't about the files you edited — it's about the files they affect. Everything reasons about the dependency graph's blast radius.
  4. Incremental over full. Don't re-scan everything every time. Tracking files remember the last processed commit so skills only reprocess what changed.
  5. Separation of concerns. Code changes and generated artifacts are different things — they land in two separate commits so history stays clean and references stay stable.

The last three — behavior-graph, behavior-runner, status — are the newest additions (the behavior layer). The project's older "seven skills" tagline predates them.

What it is not

Coherence, not enforcement

The patterns and conventions are guidelines, not rules — skills adapt them where they fit. Nothing is forced; the goal is coherence and collaboration. The one exception is the newer behavior/governance layer, where deterministic facts (a broken test link, a regressed test) can hard-block a change while model judgment (does this violate a principle?) must be resolved before proceeding — the Governance page explains how. Everywhere else, the toolkit is not prescriptive, not complete, and not perfect — and its certainty scores say so out loud.
