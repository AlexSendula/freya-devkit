Governance

Enforcing captured intent

Recording intent is only half the job — it also has to be enforced when code and specs change. Governance runs inside wrap-up, and its defining rule is simple: a failure is gated by what kind of check produced it, not by a model's self-reported confidence.

Two kinds of gate

Block on facts

Deterministic checks — a broken test link, a duplicate ID, a regressed accepted behavior — are cheap and certain. They hard-block (fail-closed): a non-zero script exit stops wrap-up until you fix it or reclassify the behavior.

Resolve to proceed

Model-judgment checks — does this violate a principle? contradict a decision? drift from a declared choice? — are not script hard-blocks. Each finding must be resolved (fixed, refuted, or amended) before wrap-up completes. They fail-open on missing inputs.

The rule

"Ignore and push" is not a resolution. Model findings aren't carried forward as backlog debt — each is resolved in the wrap-up that raised it.

The wrap-up Phase 3.5 pipeline

All of it runs in one place, in a deliberate order: deterministic facts first, then model judgment (principles → contradictions → drift). Step through it:

1 · Link & ADR integrity hard-block

Deterministic checks: every locator resolves, Gherkin tags are present, no accepted-but-TODO(scaffold), no duplicate BEH-NNN, and ADR frontmatter/links are valid. A non-zero exit blocks wrap-up.

verify_links.py --format json   &&   adr.py verify --project .

2 · Declared-intent gate (G1) hard-block

If an accepted behavior's linked test was modified or deleted in this change-set without a new INTENT-NNN record naming it, this blocks — a bare accepted-test change is a regression. With no baseline yet, the gate skips.

verify_intent.py --project . --format json   # read the JSON on non-zero

3 · Run affected accepted behaviors hard-block

Refresh the behavior graph, then re-run only the accepted behaviors whose exercised code the change touched (Direction A). A test-failed blocks until you fix the code, record an intended change, or quarantine an infra failure.

behavior_graph.py --build --project . && \
behavior_graph.py --check --base "$BASE" --project .

4 · Validate-on-hit advisory

Read-only. Surfaces the proposed/confirmed behaviors this change touched (offer to confirm them) and any touched code no behavior covers (offer to capture one). Bounded to the affected subset and fully skippable — it never changes the exit code.

behavior_graph.py --surface --base "$BASE" --project .

5 · Principle checkpoint (G2) resolve-to-proceed

Judge the diff against each principle in the constitution. Model judgment, so it's a procedural gate, not a script exit — but wrap-up must not complete while a finding is unresolved (fix / refute / amend). Prior resolutions auto-clear when still valid.

principles.py list   ·   principles.py prior/resolve …

6 · Contradiction check (G3) resolve-to-proceed

For each spec or ADR changed this cycle, check it doesn't contradict higher-authority intent (principle > ADR > spec). ADRs are compared always-global. Resolve each escalated finding before completing.

contradictions.py context --spec <ID>   ·   adr-context --adr <ID>

7 · Declarative-drift check (P4b) resolve-to-proceed

Does the changed code contradict a declared decision (a spec's intentional decisions or an ADR body)? Scoped by blast radius — deliberately not always-global like G3. Degrades to changed-files-only when code-graph is absent (never a silent empty set).

drift.py context --base "$BASE" --project .

G1 · Declared-intent records

An accepted behavior's test is its machine-checkable guarantee. So changing that test requires declaring intent: a new INTENT-NNN record, created in the same change-set, naming the behavior. Among the intent guards (G1–G3/P4b), this is the only one that is purely deterministic and hard-blocking — the others are model judgment, resolve-to-proceed. (Link and ADR integrity and the accepted-behavior run are deterministic hard-blocks too.)

  • Only accepted behaviors are governed — the others have no guarantee to protect.
  • Temporal self-scoping: the record must be new in the change-set; a past record can't bless a future edit.
  • The gate verifies a record exists, not that its rationale is honest — honesty is a separate, model-judgment track.
/freya-devkit:spec-manager intent new BEH-003
# … then edit the test; wrap-up's G1 gate now passes.

G2 · Principles (the constitution)

principles.md is the highest-authority intent record. It's enforced two ways: soft injection (surfaced when you create or scan so you draft against the rules) and a resolve-to-proceed checkpoint at wrap-up (judge the diff against each principle; resolve every finding). Empty constitution ⇒ the step skips.

G3 · Contradictions & P4a · ADRs

G3 checks that intent doesn't contradict intent: when a spec or ADR changes, compare it against higher-authority intent. P4a adds ADR support — cross-cutting decisions in decisions/. Only accepted ADRs are authoritative, and they're compared always-global (every accepted ADR, no category scoping — over-scoping would be a silent miss). Malformed ADRs surface as warnings, never silently dropped.

P4b · Declarative drift

The mirror of G3. Where G3 is intent-vs-intent, P4b is code-vs-declared-intent: does the changed code contradict a decision a spec or ADR declared? It's code-anchored — scoped by the change's blast radius (a decision's related_code intersected with the impact set), deliberately not always-global like G3. If code-graph is missing it narrows to changed-files-only and says so — never a silent empty set.

Honest limit: a declared spec or ADR with no related_code is invisible to the drift check (its intersection is always empty). The on-demand spec-manager drift gaps view lists those drift-blind items so anchors can be added.

Resolution logs

Every resolve-to-proceed decision other than a straight code fix is recorded in an append-only log at the knowledge-base/ root, so the same finding isn't re-litigated every cycle (a code fix needs no log — git records it, and the finding is gone):

LogGuard
principle-resolutions.jsonlG2
contradiction-resolutions.jsonlG3
drift-resolutions.jsonlP4b

All three share one append-only core (resolution_log.py): append a line, load latest-per-key, drop superseded entries. On the next wrap-up a prior resolution auto-clears if the flagged code is materially unchanged, is retired if the code changed out from under it, or escalates if it's now a real violation.
