Patterns & Conventions

The connective tissue

A handful of patterns recur across the skills, and a few conventions let independently-authored skills compose without hard coupling. Both are guidelines, not requirements — used where they fit.

Conventions, not requirements

There's no framework enforcing any of this. A skill that ignores a pattern still works; the patterns exist so that skills fit together naturally. "The goal is coherence and collaboration, not rigid compliance."

Coordinator + parallel workers

Problem: processing a large codebase sequentially is slow. Solution: one coordinator analyzes structure and plans the work, then spawns specialized workers that run in parallel, and their results are aggregated.

Two-commit separation

Problem: generated artifacts reference code commits; if artifacts share the commit they describe, the reference is unstable. Solution: code changes land in one commit, generated artifacts in a second.

Commit 1 — code

src/lib/auth.ts
src/api/routes.ts
tests/auth.test.ts

Commit 2 — artifacts

knowledge-base/reference/API.md
knowledge-base/specs/auth/SPEC-001.md
knowledge-base/security/.../2026-07-12.md

Cleaner history, a stable commit for the security scan to reference, and no tracking-file hacks. Used by wrap-up and codebase-security-resolver. The behavior layer refines it: a scaffold's commit follows its lifecycle state, not its location.

Incremental updates

Problem: full scans are expensive and most changes are localized. Solution: remember the last processed commit and only reprocess what changed — expanded by blast radius.

  1. Read the tracking file → last_commit
  2. git diff last_commit..HEAD --name-only → changed files
  3. If code-graph available: impact <changed> → blast radius; else the changed files
  4. Process only that set
  5. Write the tracking file with the current commit

Used by the update mode of code-graph, docs-manager, spec-manager, and codebase-security-scan.

Certainty scoring

Problem: AI-generated content isn't always correct, and humans need to know how much to trust it. Solution: inferred specs carry a 0–100 certainty score in four bands.

ScoreConfidenceAction
90–100HighAccept / trust it
70–89GoodBrief review
50–69MediumAsk the user to confirm
0–49LowDetailed review

Certainty rises with code comments, matching docs, clear patterns, and tests; it falls with no comments, ambiguity, and missing tests. User-authored specs are set to 100.

Certainty is not the trust signal for behavior

Used only by spec-manager. After the behavior layer, certainty gates review of inferred, declarative specs; trust in an executable behavior is carried by its lifecycle state (proposed → confirmed → accepted), not the number. It's a heuristic, not a formula.

More patterns

Intentional-design tracking

Specs record on-purpose decisions (and a "security scan note") so scanners reclassify them as intentional design rather than bugs. Stored by spec-manager, read by the security scan.

Fallback without dependencies

Check whether an optional teammate (usually code-graph) is available; if not, degrade to a plain git diff and warn about reduced coverage. Used by every impact-aware skill.

Phase-based execution

Break a multi-step workflow into named, reported phases ("Change Detection", "Impact Analysis", …) so progress is visible. Used by most multi-step skills.

Git-aware state

Store the last-processed commit in a dotfile so "what changed" is just a git diff away. Backs incremental updates across the toolkit.

Validation against specs

Before finalizing, cross-reference findings against specs — if the "vulnerable" behavior is explicitly allowed, mark it intentional. Used by the security scan's check-specs.

Worker specialization

Spawn domain-specialized agents rather than generic ones — one per doc type, one per security category. Sharper than a single do-everything pass.

Integration conventions

How a skill declares its place in the ecosystem so others (and the AI) understand the relationships:

ConventionWhat it means
Frontmatter declarationsTRIGGER phrases, an INTEGRATION line ("uses X when available"), and a Used by line let agents understand relationships.
scan vs updatescan/build/init = full, first-time or complete refresh. update = git-aware incremental (the day-to-day default).
Namespaced cross-referencesSkills call each other as /freya-devkit:<skill> — the bare /<skill> form only resolves for loose ~/.claude/skills/ installs.
Artifact locationsOutput goes to the right knowledge-base/ subdirectory for its type — docs to reference/, specs to specs/, findings to security/.
Dated-report idempotencyReports are written to <type>/YYYY-MM-DD.md; same date = same file (git provides history).
Tracking files.<skill>-last-<action> with a commit + timestamp, next to the skill's primary output.
Bundled script pathsScripts are referenced via ${CLAUDE_PLUGIN_ROOT} and always quoted (install paths can contain spaces).

Where the docs lag the code

The toolkit is on an active feature branch, so some prose trails the implementation. This explainer follows the shipped code as ground truth. The honest gaps worth knowing:

Prose vs. implementation
  • "Seven skills" — the README tagline predates the behavior layer; ten skills ship.
  • Artifact root — a few templates/evals still write to the old docs/ layout; knowledge-base/ is authoritative (there's a documented migration).
  • .graph/ version control — some prose calls the graph "version-controlled," but the code auto-ignores the whole cache dir. It's a regenerable local cache by default.
  • Certainty wording — the action column differs between two docs ("Trust it" vs "Accept automatically"); same bands, different phrasing.
  • behavior-graph CLI — its SKILL.md documents 3 modes; the script implements 7 (the rest are used by wrap-up, status, and the security scan).
  • Portability — several steps (≈13 in spec-manager, 2 in wrap-up) hardcode an absolute install path instead of ${CLAUDE_PLUGIN_ROOT}; the portable form is canonical (a known parking-lot item).
