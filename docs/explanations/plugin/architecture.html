Architecture

How the skills fit together

The skills form a tiered graph. One foundation skill derives a dependency graph from your code; everything above it reasons about blast radius. For the most part they don't call each other's code — they cooperate by reading and writing one shared knowledge-base/ folder. (The behavior-layer skills are the exception: they reuse spec-manager's parser in-process and run code-graph as a subprocess.)

The five tiers

Each tier depends only on the ones below it. Dependencies are optional: a higher-tier skill uses a lower one when it's installed and degrades gracefully when it isn't. Click any skill to see what it reads and writes.

code-graph, the keystone

code-graph is the foundation everything else queries. It parses import/export relationships (via regex, not a full parser) for TypeScript/JS, Python, and Go, and builds a reverse-dependency graph: for every file, who depends on it. That reverse map is what turns "the file I changed" into "everything my change affects":

Blast radius

impact(file) = file + direct_dependents(file) + transitive_dependents(file)

Computed with a cycle-safe walk over the reverse edges. Consumers ask for the impact set, then process everything in it — not just the literally-changed files.

Every import is tagged one of three ways, and the distinction is load-bearing:

TagMeaning
internal (no prefix)Real project wiring — an import that resolved to a file in your project.
external:<pkg>A third-party package (recorded as a leaf, never traversed).
unresolved:<import>Looked internal but couldn't be mapped to a real file — surfaced, not silently dropped.

An edge is internal only when it carries neither prefix. That predicate is reused elsewhere — for example, spec-manager's onboarding counts internal edges (not raw file count) to tell a real codebase from a bare scaffold.

Output lives at knowledge-base/.graph/graph.json + classifications.json. See the code-graph skill for the full schema and limits (dynamic imports, monorepos, regex parsing).

Composition: artifacts, not calls

The graph and report artifacts are the primary composition surface: most skills cooperate by leaving structured files on disk that the next skill reads — which is what keeps them independently installable and gracefully degradable. (The behavior-layer skills go further and reuse peer code directly — behavior-graph and behavior-runner import spec-manager's frontmatter parser and run their peers as subprocesses.)

The knowledge-base/ tree

Every durable artifact lives under one project-local, version-controlled root. This is the shared substrate — it travels with your branch and diffs in code review.

Committed vs regenerable

Prose and machine indexes (specs/, reference/, findings.json, BACKLOG.md, the resolution logs) are meant to be committed. The .graph/ cache is regenerable: both graph tools (code-graph and behavior-graph) auto-write a .gitignore containing * into it, so by default the whole graph directory is git-ignored — even though some prose calls the graph "version-controlled." Treat .graph/ as a local cache; rebuilding is cheap.

Graceful degradation

The contract every impact-aware skill honors: use code-graph if it's available, otherwise fall back to a plain git diff of the directly-changed files — and say so.

code-graph available

impact = code-graph impact <changed>
# process the full blast radius:
# changed files + their dependents

code-graph absent (fallback)

files = git diff <last>..HEAD
# process directly-changed files only,
# warn: "reduced coverage"

This is why the skills "work standalone but work better together." No skill hard-requires another; missing dependencies narrow coverage rather than breaking the run.

Incremental updates & tracking

Full scans are expensive; most changes are localized. So each skill records the last commit it processed in a tracking dotfile and, next time, only reprocesses git diff <last>..HEAD (expanded by blast radius).

Never-synced guard

Incremental update assumes a prior sync. On a project with no tracking files, wrap-up must not let update silently trigger a full-codebase generation — it reports the project is unsynced and defers to an explicit first-time scan / build. Running spec-manager bootstrap first is what makes the first wrap-up behave.

Where the behavior layer sits

The behavior layer adds a second graph on top of the first. behavior.json is a sibling of graph.json under .graph/. It projects your specs' behavior records and links them through tests down to the code they exercise — a SPEC → BEHAVIOR → TEST → CODE chain — then answers blast-radius questions in both directions:

→

→

behavior-graph stays a pure graph layer: it queries code-graph for impact and behavior-runner for coverage; code-graph never learns about behaviors. The full mechanics live on the Behavior Layer and Governance pages.
