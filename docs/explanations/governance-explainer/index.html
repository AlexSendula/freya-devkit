Governance track · Phase 3

Governance, and the ADR question

Why the contradiction checker (G3) doesn't yet look at architecture decisions — and why that's a missing room, not a missing intent.

G1 declared-intent · shipped G2 principle enforcement · shipped G3 contradiction checks · in design
01 · The track

Three ways to keep intent honest

The governance track makes the project's recorded intent enforced, not decorative. Three sub-projects, each guarding a different seam.

G1 · SHIPPED

Declared-intent records

You can't silently change an accepted behavior's test. Editing it needs an INTENT-NNN record saying why. A deterministic gate — a hard fact.

G2 · SHIPPED

Principle enforcement

At wrap-up, the code change is judged against principles.md. Each finding is fixed / refuted / amended before you finish. Model judgment — resolve-to-proceed.

G3 · IN DESIGN

Contradiction checks

When a spec's intent changes, does it contradict a higher-authority intent? Intent-vs-intent, resolved by the authority order below. This is the page you're reading about.

The distinction that matters

G1 & G2 compare against code (a test, a diff). G3 compares one recorded intent against another recorded intent. So the question "which intents does G3 know about?" is the whole ballgame — and it's where ADRs come in.

02 · Authority

Intent has a chain of command

When two intents disagree, the higher one wins (or must be consciously amended). G3 resolves every contradiction against this order.

Highest

principles.md — the constitution. Project-wide rules. exists

▲ wins over ▼
Middle

specs/ (per-feature intent + inline decisions)  and  decisions/ (cross-cutting ADRs). Same authority level as each other. specs exist ADRs: empty room

▲ wins over ▼
Lowest

reference/ — describes how the code currently is. Reverse-synced, never authoritative.

Notice the middle rung holds two things. One of them — specs/ — is fully built. The other — decisions/, the ADRs — is an empty room. That single fact is the whole confusion.

03 · The crux

"Decisions" means two different things

This is almost always the source of the confusion. There are two kinds of decision in this system, and G3 treats them very differently — because only one of them physically exists yet.

✓ Exists today (since Phase 1)

Inline spec decisions

The ## Intentional Design Decisions a spec carries — tied to that one feature.

  • "No password fallback — it's a phishing vector."
  • "Uniform 404 to prevent user enumeration."

G3 checks these. They're half of its comparison set (same-category peers).

✗ No machinery yet

ADRs — Architecture Decision Records

Cross-cutting decisions that aren't tied to one feature — they constrain the whole project.

  • "We use Postgres (ADR-007)."
  • "Multi-tenancy is row-level, not schema-per-tenant."

G3 can't check these yet — there's nowhere to write them and no format to read them.

So, to be precise

When the design says G3 is "ADR-blind," it means only the right-hand box. G3 does govern the left-hand box (per-feature spec decisions) and the constitution (principles.md). It's not blind to "decisions" — it's blind to the one kind of decision that has no home yet.

04 · The real reason

You can't check against an empty room

ADRs aren't excluded because they don't matter. They're excluded because the thing that would hold them hasn't been built. Look at the actual folder:

knowledge-base/
  principles.md            ← constitution — real rules inside     ✓ built
  specs/auth/SPEC-001…     ← per-feature intent + decisions       ✓ built
  decisions/              ← ADRs live here…                      ✗ empty scaffold
    (a .gitkeep, and nothing else)
  principle-resolutions.jsonl ← G2's audit log                    ✓ built

The decisions/ folder is created by spec-manager init, but it's a placeholder. There is:

To govern ADRs, you'd need…Status
An ADR file format — a parseable ADR-NNN.md (decision, rationale, rejected alternatives, revisit conditions)doesn't exist
A create flow — adr create, ID allocation, indexing (like spec create)doesn't exist
A way to gather them — so the checker can load "all ADRs in this domain"doesn't exist
The honest version

If G3 tried to check specs against ADRs today, it would be checking against nothing — an always-empty set — while looking like it covers architecture decisions. That false sense of coverage is worse than an honest gap. So G3 is built ADR-ready: the same check extends to ADRs the moment the machinery lands. It's a wired-but-unplugged socket, not a wall.

05 · Coverage

What G3 catches now vs. what waits

The gap is real and worth seeing plainly. Here's exactly which conflicts G3-as-scoped catches, and which need the ADR machinery first.

A change that…ExampleG3 now?
contradicts a principlenew spec adds a public endpoint; principle says "authenticated by default"✓ caught
contradicts a same-feature-area spec decisiona new auth spec says "password fallback allowed"; a peer says "never — phishing vector"✓ caught
contradicts a cross-cutting architecture decisiona spec assumes Mongo; the (unwritten) ADR chose Postgres✗ needs ADRs
Not vacuous

Two of three real conflict classes are covered on day one. "New intent violates the constitution" and "two features quietly disagree" are exactly the everyday drift a solo/AI workflow produces. The third class — silent violation of an architecture decision — is precisely what the ADR sub-project unlocks next.

06 · Your call

So — what do we do about ADRs?

This is a genuine sequencing decision, not a settled one. Three honest paths:

◆ Recommended

G3 now, ADRs next

Ship G3 against principles + spec decisions (real coverage), designed to extend. Then build ADR capture as its own sub-project — after which the same G3 check gains ADR-awareness for free. Each piece stays clean and testable.

Bigger

ADR machinery first

Pause G3. Build a minimal ADR format + adr create + index in decisions/. Then G3 covers principles + spec decisions + ADRs from day one. ADRs land sooner; the contradiction check waits.

Middle

Fold a minimal ADR format into G3

Add just enough ADR structure (a parseable ADR-NNN.md + a gather step) so G3 covers ADRs — without a full create/lifecycle flow. Faster, but couples two concerns and blurs G3's boundary.

Bottom line

Nothing about "we don't care about architecture decisions" is true here. ADRs were always in the plan (vision Phase 4). The only question is order: finish the contradiction engine now and plug ADRs into it next, or build the ADR room first so nothing about "decisions" ships half-covered.
